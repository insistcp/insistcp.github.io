计算机网络学习笔记-计算机之间是如何通讯的

发表于 2017-08-26 | | 阅读次数

1：问题提出：现实生活中的两台电脑是如何进行通讯的？



如上图所示：小明ip：10.1.1.2/24 ,小红ip：10.1.1.3/24他们俩要相互通讯。
请求流程图

  • 1）：小明先判断两个ip地址是否在同一个网段，怎么判断是否在同一个网段呢？看两个IP地址的子网掩码都是10.1.1所以两个ip在同一个网段，此时小明就要查询自己电脑的ARP缓存看是否有小红电脑的MAC地址，如果有拿出来，直接将要发送的信息由mac地址直接发送即可。如果没有则需要查询发送一个arp广播获取小红电脑的mac地址（arp广播如果交换机没有划分vlan则在所有端口都会进行一个arp请求发送），拿到mac地址之后双方便可进行通讯
  • 2）：如果两台电脑不在同一个网段： 小明ip：10.1.1.2/24，小丽10.1.2.1/24;
    先判断两个ip是否在同一个网段，不在，此时查询小明的路由表，看看是否有和小丽在同一个网段的路由记录，如果有由对应的端口号发出到其下一跳地址，一样得先获取其mac地址然后再根据mac地址发送报文。如果没有匹配到路由记录则走默认路由，0.0.0.0下一跳就我们配置地默认网关10.1.1.1。一样我们得先获取到默认网关得mac地址，大家仔细观察会发现，默认网关和小明所在ip在同一个网段，所以只需要发送arp请求就可以获取到mac地址。网关硬件接口受到消息之后，将消息放到缓冲区，并通知ip进程过来取，网关的Ip进程从网卡那里拿到ip包之后，接下来就要进行路由匹配查找(采用路由最长匹配原则，根据具体情况发送请求获取目的端得mac地址)。小红受到请求之后按照之前得请求流程将回应包发送给小明。

2：问题提出：当我们访问外网时，网络具体是怎么运作得？

还是那上面的那个例子将，现在小明想要访问外网（www.cpinsist.xyz）；流程该怎么走呢？

  • a：首先小明在浏览器里面输入www.cpinsist.xyz这个地址，但是我们都知道网上的电脑想要相互通讯他们就必须知道彼此的ip地址，所以现在浏览器首先要将www.cpinsist.xyz这个地址转换成ip地址，
    首先浏览器会告诉本机的DNS进程让它先在自己的缓冲中查找，看看是否有该地址对应的缓存ip，如果没有则查询本机的host文件，看看是否有对应的地址和ip。如果没有，则检查本地DNS服务器的配置，如上图所示10.10.10.10.让它帮忙查找，此消息是以UDP的格式进行传输的，小明发现不在同一个网段，就采用上面提到的不同网段之间的通信。将IP包发给DNS server。如果此时还是没有找到，于是向上级DNS server（8.8.8.8）发送协查请求。此时返现该地址对应的IP地址，将结果依次返回到小明电脑的DNS进程。
  • b:接下来 ：小明的浏览器将小明发送的请求打包成HTTP的格式，并通知TCP进程，同时告诉TCP对应的IP地址，让TCP去完成信息传递的任务。
    此时为了安全考量，TCP进程在向目标地址发送信息之前会先发一个IP包确定可以到达目的服务器，并且目的服务的IP包也可以安全返回的前提下再发送HTTP请求。即：HTTP请求的3次握手，小明的TCP进程先目的IP发送请求，询问对方是否收得到自己的消息，目的主机收到消息后向小明回传一个消息（告诉小明自己能手受到消息，同时询问小明是否能够收到自己发送的消息）小明接受到消息后会再发送一个消息告诉目标主机自己可以收到消息。
    确定双方之间可以正常往返消息后，小明便开始发送http请求，小明的TCP进程使用TCP头将http打包发送给IP进程，IP进程在加上自己的头部信息，在IP进程眼里这个仅仅就是一个IP包而已，那么问题又来了，怎么知道对方是否接受了这个IP包呢？这就需要TCP的确认机制，对方在收到小明发送的消息后会给小明返回一个确认信息，告知小明自己已经收到消息。另外一个问题就是小明的TCP进程怎么知道对方收到的是哪个IP包，小明的TCP进程在发送TCP消息之前会将每一个字节打上标记Sequence Number（例如小明想要传送100个字节的数据，初始序列号从1开始，那么每一个字节的数据都对应有一个Sequence number）。对方在收到TCP消息之后，会根据根据最后收到的字节返回一个Acknowledage number 在这里这个数字为101，告诉小明的TCP进程，前100个字节的数据已经接收到，现在等着接收第101字节的数据。此时 www.cpinsist.xyz对应的服务器就开始向小明回传数据，同样是将数据封装成http格式，小明在接收到数据之后回传确认收货的消息（同样是以IP+TCP+http的数据包），最后就是释放连接，将小明和服务器之间的连接看成是两根相互连接的管道，管道1 ：对小明而言：小明向服务发送消息的管道，对服务器：接收小明发送消息的管道，管道2：对小明：小明接收服务器回传消息的管道，对服务器：自己向小明发送消息的管道；当小明已经没有消息向服务器请求时，向服务器发送一个FIN+ACK+Squ的 数据包，自己进入FIN_wait1等待状态，这时服务器接收到小明的请求之后，告诉小明，我收到你的关闭请求，于是向小明发送一个ACK+squ消息报，在此之后自己便进入了close_wait装态，小明在接收到这个消息之后便进入FIN_wait2状态，服务器等待自己将所需要传输的的数据都传输完毕时，再向小明发送一个FIN+ACK+Sqe的数据包之后自己进入last_ack阶段，小明在收到这个数据包之后就知道，服务端已经关闭了通道2，所以小明先确定已经收到了对方发来的全部数据，之后先要给对方回复一个ACK的数据报告诉对方自己已经收到了关闭请求数据，此时小明处于time_wait状态，而服务端在接受到小明的确认之后便进入close状态；而小明还要等一会直到自己通道1中没有数据传输之后便自动关闭通道1，等待时间一般为2MSL;
